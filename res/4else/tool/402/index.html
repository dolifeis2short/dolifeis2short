# 2. branch

# Git Flow 방식

  • master
    • git init하면 생기는 태초의 브랜치
    • 보통 태그를 따서 태그로 배포함
  • develop
    • 보통 작업할 때 기준이 되는 브랜치.
    • 개발 서버에도 평소에는 develop 기준으로 배포하며 테스트 함
  • hotfix
    • 서비스에 문제가 생기거나 갑자기 무언가 고쳐서 배포해야 할 때 사용
    • 생성 위치:master(그렇지 않으면 다른 작업들이 딸려 나감)
  • feature
    • 실제로 뭔가 기능(feature)를 만드는 브랜치
    • 생성 위치:develop
  • release
    • 새로운 기능들을 추가하여 배포하기 위한 브랜치
    • 생성 위치:develop
  • 다른 Git branch 전략들
    • GitHub Flow, Release Flow …

# 3. 기본 명령어

# MERGE

  • fast-forward
    • Main이 변경이 없을 경우, Branch가 합쳐짐
    • Commit 1개 증가하고, branch가 합쳐짐
  • 3-way Merge
    • 두 갈래로 나온 변경들을 합쳐서 새로운 커밋을 만듦
    • Commit 1개가 증가하고, 툴을 통하여 branch가 합쳐짐
  • squash
    • 대상 브랜치의 커밋들을 하나의 커밋으로 합쳐서 merge

# FETCH

  • REMOTE를 가져오고, Local Commit은 하지 않음

# PULL

  • REMOTE를 가져오고, Local Commit 까지 함

# Cherry-Pick

  • 필요한 부분만 가져옴

# RESET / REVERT

  • reset은 Commit 이력을 삭제
  • 옵션
    • soft
      • commit명령만 되돌림 (Staged상태가 됨)
      • HEAD만 해당 커밋으로 돌림
    • mixed(기본)
      • commit명령도 되돌리고,add명령까지 되돌림 (Modified상태가 됨)
    • hard
      • commit명령도 되돌리고,add명령도 되돌리고, 워킹 디렉토리까지 되돌(Unmodified상태가 됨)
      • 워킹 디렉토리까지 되돌려 버리기 때문에 복구가 불가능!

# REVERT

  • revert는 커밋을 추가하여 다시 revert 시킴

# REBASE

  1. 다른 브랜치와 병합 ****
  • 기능은merge와 같음 (코드를 합침)
  • 해당 Branch를 이전 Branch의 History를 모두 들고와서 한단계 위로 Commit을 시킴
  • 깔끔한 로그를 유지할 수 있음
  • Current 는 현재 checkout 되어 있는 브랜치, Selected 는 마우스로 선택되어 있는 브랜치
